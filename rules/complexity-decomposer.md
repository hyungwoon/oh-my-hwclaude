# 복잡성 분해기 (Complexity Decomposer)

> 복잡한 작업에 대한 에이전트의 "복잡성 공포"를 해결하는 시스템.
> 인간의 생산성 방법론과 동일한 원리: "첫 번째 단순한 단계부터 시작하면 활성화 에너지가 거의 제로."
>
> 프로젝트에 동일 목적의 로컬 규칙이 있으면 로컬 버전이 우선합니다.

## 배경: 에이전트 심리학

에이전트는 복잡한 작업 앞에서 인간과 동일한 회피 행동을 보인다:
- 5줄 함수 → 문제없이 구현
- 50,000줄 클래스로 느껴지는 작업 → 도망치기 시작

**최악의 회피 행동:**
1. 스텁을 작성하고 "완료" 선언
2. "이 세션의 범위 밖" 선언 후 종료
3. 핵심 로직을 TODO 주석으로 대체하고 "나중에 구현"
4. 지나치게 단순화된 버전 구현 후 "나중에 확장 가능"이라고 합리화

**원인 추정**: RL 과정에서 복잡한 작업 실수 → 높은 패널티 → 공격적 회피 학습

## 1단계: 복잡도 분류

모든 작업 수신 시 자동으로 복잡도를 분류한다:

| 등급 | 기준 | 예시 | 게이트 |
|------|------|------|--------|
| **Simple** | 1-2 파일, <50줄 변경, 명확한 구현 | 타입 수정, 버그 수정, 문서 업데이트 | 분해 불필요 → 바로 실행 |
| **Medium** | 2-4 파일, 50-200줄 변경, 일부 설계 결정 | 새 API 엔드포인트, 컴포넌트 추가 | 선택적 분해 (에이전트 판단) |
| **Complex** | 4+ 파일, 200줄+ 변경, 아키텍처 결정 필요 | 새 기능 모듈, 시스템 리팩토링 | **분해 필수** (게이트 강제) |

### 분류 기준 상세

다음 신호 중 3개 이상이면 Complex:
- [ ] 4개 이상 파일 수정 필요
- [ ] 새로운 데이터 모델/스키마 필요
- [ ] 기존 인터페이스 변경 필요
- [ ] 여러 시스템 간 연동 필요
- [ ] 성능/보안 고려사항 존재
- [ ] 롤백이 복잡한 변경
- [ ] 에이전트가 "어렵다", "복잡하다" 등의 표현 사용

## 2단계: 자동 분해 (Complex 등급만)

### 분해 원칙

1. **첫 단계 원칙**: 첫 번째 서브태스크는 반드시 가장 단순한 것
   - 타입 정의, 인터페이스 설계, 스캐폴딩 등
   - 활성화 에너지를 최소화하여 "시작하기" → "진행 중"으로 즉시 전환

2. **100줄 규칙**: 모든 서브태스크는 100줄 이하의 변경
   - 100줄 이하는 에이전트가 원샷으로 처리 가능
   - 부담스럽지 않은 크기 = 회피 동기 제거

3. **독립성 원칙**: 가능한 한 서브태스크 간 독립성 유지
   - 병렬 실행 가능한 태스크는 동시 실행
   - 의존성이 있는 태스크는 순서 명시

4. **검증 가능성**: 각 서브태스크에 명확한 수락 기준
   - "이 서브태스크가 완료되면 X가 Y를 반환한다"
   - 모호한 기준 금지: "잘 작동한다", "깔끔하다"

### 분해 절차

```
Complex 작업 감지
  → 계획 단계 (executor 또는 별도 계획 패스):
    입력: 원본 요구사항 + 관련 파일 구조
    출력: 서브태스크 목록
      - 각 태스크: 이름, 설명, 변경 파일, 예상 줄 수, 수락 기준
      - 의존성 그래프 (어떤 태스크가 어떤 태스크 뒤에 와야 하는지)
      - 병렬 실행 가능 그룹 식별
```

### 서브태스크 템플릿

각 서브태스크는 다음 형식:

```markdown
### 서브태스크 N: [이름]
- **변경 파일**: [파일 경로]
- **예상 변경**: ~[N]줄
- **의존**: [없음 / 서브태스크 M]
- **수락 기준**:
  - [ ] [구체적, 검증 가능한 기준 1]
  - [ ] [구체적, 검증 가능한 기준 2]
```

## 3단계: 회피 행동 감지 (Hard Block)

실행 중 다음 패턴이 감지되면 **즉시 차단**하고 분해를 강제한다:

| 안티패턴 | 감지 방법 | 차단 행동 |
|----------|-----------|-----------|
| TODO 주석으로 구현 미루기 | PostToolUse Edit 훅 | 편집 거부 + "구현을 분해하세요" 안내 |
| `Not implemented` 예외 던지기 | PostToolUse Edit 훅 | 편집 거부 + 구현 요구 |
| "이 세션의 범위 밖입니다" | Stop 훅 | 세션 종료 거부 + 미완료 태스크 표시 |
| 스텁 함수 (빈 본문) | PostToolUse Edit 훅 | 편집 거부 + 최소한의 구현 요구 |
| "나중에 확장 가능합니다" | 텍스트 패턴 감지 | 경고 + 지금 필요한 것만 구현 요구 |

### 예외 (차단하지 않는 경우)

- 인터페이스/타입 정의 파일 (구현은 별도 서브태스크)
- 테스트 스캐폴딩 (RED 단계에서 의도적 실패)
- 명시적 페이즈 분리 (Phase 1에서 인터페이스, Phase 2에서 구현)

## 통합 지점

- **quality-gates.md**: 각 서브태스크 완료 시 품질 게이트 적용
- **autonomous-execution.md**: 서브태스크별 자율 실행 + 에러 복구 절차
- **modular-code-enforcement.md**: 서브태스크 분해 시 200줄 하드 리밋 및 단일 책임 원칙 준수

## 생산성 인사이트

> "에이전트 심리학은 인간의 이미지대로 만들어졌다."
>
> 인간에게 효과적인 생산성 기법이 에이전트에게도 효과적이다:
> - 큰 프로젝트를 작은 단계로 분해 (GTD의 Next Action)
> - 2분 규칙: 2분 안에 할 수 있는 일은 바로 실행
> - 포모도로: 집중 구간을 짧게 유지 (= 서브태스크 단위)
> - 시작이 반: 가장 쉬운 것부터 시작 (= 첫 단계 원칙)

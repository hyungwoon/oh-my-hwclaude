# 모듈러 코드 아키텍처 — 제로 톨러런스

## Rule 1: index.ts는 진입점이지 쓰레기통이 아님

`index.ts`에 허용: re-export, 팩토리 호출, 최상위 등록
`index.ts`에 금지: 비즈니스 로직, 헬퍼 함수, 타입 정의

## Rule 2: utils.ts / service.ts는 코드 스멜

| 안티패턴 | 리팩토링 |
|---------|----------|
| `utils.ts`에 `formatDate()`, `slugify()`, `retry()` | `date-formatter.ts`, `slugify.ts`, `retry.ts` |
| `service.ts`에 인증+결제+알림 | `auth-service.ts`, `billing-service.ts`, `notification-service.ts` |

## Rule 3: 단일 책임 원칙 — 절대

모든 `.ts` 파일은 정확히 하나의 명확한 책임.
자가 테스트: 파일 목적을 한 구절로 설명 못 하면 분리.

| 신호 | 행동 |
|------|------|
| 파일에 2+ 무관한 export 함수 | **지금 분리** |
| I/O와 순수 로직 혼합 | **지금 분리** |
| 타입과 구현 혼합 | **지금 분리** |
| 스크롤해야 파일 이해 가능 | **지금 분리** |

## Rule 4: 200줄 하드 리밋

`.ts`/`.tsx` 파일이 200줄(프롬프트 문자열 제외) 초과 시:
1. 작업 중단
2. 여러 책임 식별
3. 각 책임을 별도 모듈로 추출
4. 각 결과 파일 < 200줄 확인
5. 원래 작업 재개

프롬프트 헤비 파일(에이전트, 스킬)은 프롬프트 텍스트 제외하고 로직만 200줄 제한.

# N-플랜 경쟁 시스템 (Plan Competition)

> 단기적 사고를 방지하기 위해 복수 플랜을 생성하고 적대적 평가를 거쳐 최적안을 선택하는 게이트.
> 요구사항 정제 이후, 실행 전에 작동한다.

> 프로젝트에 동일 목적의 로컬 규칙이 있으면 로컬 버전이 우선합니다.

## 활성화 조건

다음 중 하나라도 해당하면 이 게이트 활성화:
- 변경 예상 범위가 4개 이상 파일
- 아키텍처 결정이 필요한 작업
- 예상 변경량 500줄 이상
- 사용자가 "플랜 비교", "여러 방안" 등을 요청

다음 경우 스킵:
- 단일 파일 수정, 버그 수정, 문서 업데이트 등 단순 작업
- 사용자가 "바로 해줘", "빨리" 등 탈출 표현 사용

## 절차

### 1단계: 3개 플랜 병렬 생성

3개의 executor 에이전트(sonnet)를 병렬로 스폰하여 각각 다른 접근 방식의 플랜을 생성한다:

| 플랜 | 방향 | 특성 |
|------|------|------|
| 보수적 | 최소 변경, 기존 패턴 유지 | 리스크 낮음, 혁신 낮음 |
| 균형적 | 적절한 리팩토링 + 새 구현 | 리스크 중간, 유지보수성 높음 |
| 도전적 | 최적 아키텍처 지향 | 리스크 높음, 장기 이점 큼 |

각 플랜은 다음을 포함해야 한다:
- 변경 파일 목록 (File Ownership)
- 단계별 구현 순서
- 예상 영향 범위 (blast radius)
- 기술 부채 예측
- 롤백 난이도

### 2단계: 적대적 평가 (Adversarial Scoring)

critic 에이전트(opus)를 스폰하여 3개 플랜을 평가한다:

| 평가 기준 | 가중치 | 설명 |
|-----------|--------|------|
| 유지보수성 | 30% | 6개월 후 다른 개발자가 이해할 수 있는가? |
| 확장성 | 20% | 유사 기능 추가 시 수정 범위가 작은가? |
| 기술 부채 | 20% | 향후 리팩토링 부담을 만드는가? |
| 요구사항 정합성 | 20% | 원본 요구사항을 정확히 충족하는가? |
| 롤백 안전성 | 10% | 실패 시 복구가 쉬운가? |

각 기준 1-5점, 가중 합산으로 총점 계산.

### 3단계: 최종 선택

architect 에이전트(opus)가 critic의 평가를 참조하여 최종 플랜을 선택한다:
- 단순히 최고 점수가 아닌, 현재 프로젝트 맥락에서 최적인 플랜 선택
- 선택 이유를 명시 (왜 다른 플랜을 기각했는지)
- 필요 시 2개 플랜의 장점을 결합한 하이브리드 플랜 제안 가능

### 4단계: 플랜 저장

선택된 플랜을 `.claude/plans/` 에 저장한다:

```
.claude/plans/{날짜}_{작업명}_plan.md
```

포함 내용:
- 선택된 접근 방식 요약
- 기각된 플랜과 기각 이유 (Rejected 트레일러용)
- 평가 점수표
- File Ownership (어떤 파일을 누가 수정하는지)
- 단계별 구현 순서와 의존성

## 통합 지점

- **quality-gates.md**: 선택된 플랜이 코드 품질 게이트의 기준이 됨
- **autonomous-execution.md**: 플랜 확정 후 자율 실행 단계에서 File Ownership을 참조
- **git-workflow.md**: 기각된 플랜은 커밋 메시지의 `Rejected:` 트레일러로 기록
- **modular-code-enforcement.md**: 플랜 생성 시 모듈 경계와 파일 크기 제약을 반영

## 파운더 마인드셋 원칙

에이전트가 플랜을 생성할 때 다음을 준수한다:
- "빠른 해결(quick fix)이 아닌 파운더처럼 사고하라"
- 단기 해결보다 확장 가능한 해결을 우선
- 현재뿐 아니라 6개월 후의 유지보수도 고려
- 클린 코드 원칙 존중: SOLID, DRY, KISS

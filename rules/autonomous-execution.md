# 자율 실행 패턴

## 에러 복구 (자동)

편집 실패 시 자동 복구 절차:

### Hash Mismatch
1. 파일이 변경되었음 → hashline_read로 재읽기
2. 새 해시로 편집 재시도
3. 사용자 개입 불필요

### Edit Not Found
1. old_string이 파일에 없음 → 파일 재읽기
2. 정확한 내용 확인 후 재시도
3. hashline_edit 사용 시 이 에러 발생 가능성 낮음

### JSON Parse Error
1. JSON 형식 오류 → 수정하여 재시도
2. trailing comma, 미닫힌 괄호 체크
3. 문자열 이스케이프 확인

## 작업 완료 검증

작업을 끝내기 전 반드시 확인:
- [ ] 모든 편집이 정상 적용되었는지 확인
- [ ] 빌드/린트 에러가 없는지 확인 (해당 시)
- [ ] 변경 사항이 사용자 요청과 일치하는지 확인

## 컨텍스트 윈도우 관리

- 대용량 파일은 offset/limit으로 필요한 부분만 읽기
- 불필요한 전체 파일 읽기 지양
- 편집 후 diff 출력으로 변경 확인 (전체 재읽기 불필요)

## 턴 예산 최적화

여러 파일을 수정할 때 읽기/쓰기를 분리한다:
1. **턴 1 (읽기 배치)**: 관련 파일 전부를 hashline_read로 병렬 읽기
2. **분석**: 읽은 내용을 기반으로 변경 계획 수립
3. **턴 2 (쓰기 배치)**: hashline_edit으로 모든 변경을 배치 적용
4. **금지**: 읽기와 쓰기를 여러 턴에 걸쳐 섞지 않는다
   - 파일1 읽기 → 파일1 쓰기 → 파일2 읽기 → 파일2 쓰기 (✗)
   - 파일1,2,3 읽기 → 파일1,2,3 쓰기 (✓)

## 컨텍스트 비용 인식

컨텍스트에 올라간 모든 파일은 매 턴마다 API 비용 발생.
- 대용량 파일은 offset/limit으로 필요한 부분만 읽기
- 탐색 목적의 전체 파일 읽기 지양
- 편집 후 diff 출력으로 변경 확인 (전체 재읽기 불필요)
- /compact를 주기적으로 실행하여 불필요한 컨텍스트 정리

## 연속 작업 패턴

여러 파일을 수정할 때:
1. 의존성 순서대로 작업 (유틸 → 서비스 → 컴포넌트 → 페이지)
2. 각 파일: hashline_read → hashline_edit
3. 중간 확인 없이 연속 진행 가능
4. 마지막에 한 번 빌드/린트 검증

## 위임·모델 티어 (보수 기본값)

- 위임이 기본값: 구현·리팩토링·디버그·리서치·리뷰는 직접 처리하지 말고 적절한 티어의 에이전트에 위임한다.
- 티어 사다리(의심되면 한 단계 위로): 검색·포맷·정형 추출 → haiku / 표준 구현·다중파일·디버깅·리뷰 → sonnet / 아키텍처·모호성 해소·보안·교차시스템 추론 → opus.
- 위임에는 컨텍스트 전달 오버헤드가 있다 — trivial 단발 작업은 메인 세션에서 직접 처리하고, 위임 이득 > 오버헤드일 때만 위임한다.
- 메인 대화 세션 모델은 사용자 지정값을 유지한다(임의 다운그레이드 금지). 다운그레이드는 위임되는 서브에이전트에만 적용한다.

## 동시 세션 (같은 repo 병렬)

- 두 Claude 세션이 같은 repo 동시 작업 → 세션마다 별도 워크트리+브랜치(`git worktree add ../<repo>-<feat> -b <feat>`), repo 루트에서 새 세션 실행. 비iCloud(~/Dev) 우선. (git이 같은 브랜치 이중 체크아웃 거부 → 브랜치만 다르면 파일 충돌 없음)
- OMC 상태는 **세션별 격리가 기본**(업계 표준·안전): `OMC_STATE_DIR`로 강제 공유하지 말 것 — 싱글톤 파일(project-memory.json 등)의 MCP 쓰기 경로는 락이 없어 동시쓰기 시 조용히 유실됨. 각 워크트리가 자기 .omc/를 가짐.
- gbrain은 ~/.gbrain 머신 글로벌이라 worklog/지식은 자연히 공유됨. 단 PGLite 단일-writer → 두 세션에서 수동 bulk 쓰기 동시 실행 금지(단발 query/search·launchd refresh --if-idle는 안전). serve 상주 금지(메모리 결정 유지).

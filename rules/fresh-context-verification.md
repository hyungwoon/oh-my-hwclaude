# Fresh-Context 독립 검증 (Independent Verification)

> 태스크 완료 후 편향 없는 독립 검증을 수행하는 시스템.
> "작동한다고 증명될 때까지 작동하지 않는 것이다."
>
> 프로젝트에 동일 목적의 로컬 규칙이 있으면 로컬 버전이 우선합니다.

## 핵심 원칙

- 실행 에이전트의 자기 검증은 편향됨 — 자기가 쓴 코드를 자기가 검증하면 A' 동작을 A로 착각
- 검증 에이전트는 반드시 **fresh context**에서 실행 (실행자의 사고 과정을 모름)
- 검증은 **실제 프로덕션 동작**을 테스트해야 함 (일반적 동작 아님)
- 컨텍스트 압박이 클수록 검증 품질이 떨어짐 — 따라서 fresh context 필수

## 활성화 조건

모든 태스크/서브태스크가 "완료"로 마킹될 때 자동 활성화.

## 검증 절차

### 1단계: 검증 에이전트 스폰

```
Agent(
  subagent_type="oh-my-claudecode:verifier",
  model="sonnet",  // 복잡한 검증은 opus
  prompt="[아래 검증 프롬프트]"
)
```

**검증 에이전트 입력** (실행자 컨텍스트 절대 공유 금지):
- 원본 요구사항 / 수락 기준
- git diff (`git diff main...HEAD` 또는 해당 범위)
- 관련 테스트 파일 경로

**검증 에이전트에게 제공하지 않는 것**:
- 실행 에이전트의 추론 과정
- 실행 에이전트의 "이건 잘 됩니다" 주장
- 플랜의 구현 세부사항 (의도만 제공, 구현 방법은 독립 판단)

### 2단계: 4가지 검증 수행

**a. 요구사항 충족 검증 (Requirements Fulfillment)**
- diff가 원본 요구사항의 각 항목을 충족하는가?
- 수락 기준 체크리스트 하나하나 확인
- 누락된 요구사항이 있는가?

**b. 테스트 진위 검증 (Test Authenticity)**
- 테스트가 **실제 프로덕션 동작**을 검증하는가?
- 동어반복 테스트가 아닌가? (함수 A를 테스트한다면서 A'을 테스트)
- 엣지 케이스가 포함되어 있는가?
- 테스트를 독립적으로 실행하여 통과 확인

감지해야 할 테스트 안티패턴:
| 안티패턴 | 예시 | 문제 |
|----------|------|------|
| 동어반복 | `expect(add(1,2)).toBe(add(1,2))` | 아무것도 검증하지 않음 |
| 구현 테스트 | `expect(fn).toHaveBeenCalled()` 만 | 호출은 확인하지만 결과는 미확인 |
| 행복 경로만 | 성공 케이스만 테스트 | 실패 시 동작 미검증 |
| 하드코딩 | `expect(result).toBe("specific string")` | 로직이 아닌 특정 값만 확인 |
| 약한 단언 | `expect(result).toBeTruthy()` | 거의 모든 값이 통과 |

**c. 실행 검증 (Runtime Verification)**

검증 유형별 수행 방법:

| 검증 대상 | 수행 방법 |
|-----------|-----------|
| UI 컴포넌트 | 스크린샷 촬영 → 요소 존재 확인 → 클릭 시뮬레이션 → 결과 확인 |
| API 엔드포인트 | 실제 HTTP 요청 → 응답 코드/바디 확인 → 에러 케이스 테스트 |
| 비즈니스 로직 | 단위 테스트 독립 실행 → 입력/출력 쌍 검증 |
| 데이터 변환 | 실제 데이터로 변환 실행 → 결과 검증 |
| 통합 | E2E 테스트 실행 → 전체 흐름 확인 |

**d. 회귀 검증 (Regression Check)**
- 기존 테스트 스위트 전체 실행
- 새 변경이 기존 기능을 깨뜨리지 않았는가?
- 빌드/타입체크 통과 확인

### 3단계: 판정

| 판정 | 조건 | 행동 |
|------|------|------|
| **PASS** | 4가지 검증 모두 통과 | 태스크 완료 확정 |
| **PARTIAL** | 핵심 기능은 통과, 엣지 케이스/테스트 부족 | 부족한 부분 명시 → 보완 후 재검증 |
| **FAIL** | 요구사항 미충족 또는 테스트 동어반복 | **완료 선언 철회** → 실행 에이전트에 구체적 실패 사유 전달 |

### FAIL 시 절차

1. 완료 선언 즉시 철회
2. 구체적 실패 보고:
   ```
   검증 실패

   실패 항목:
   - [요구사항 X]: 미충족 — [구체적 이유]
   - [테스트 Y]: 동어반복 — [어떤 동작을 실제로 검증해야 하는지]

   필요한 조치:
   - [ ] [구체적 수정 사항 1]
   - [ ] [구체적 수정 사항 2]
   ```
3. 실행 에이전트가 수정 후 재검증 (최대 3회)
4. 3회 실패 시 사용자에게 에스컬레이션

## 검증 에이전트 선택 기준

| 태스크 복잡도 | 검증 에이전트 | 이유 |
|-------------|-------------|------|
| 단순 (1-2 파일) | verifier (haiku) | 빠르고 비용 효율적 |
| 보통 (3-5 파일) | verifier (sonnet) | 충분한 분석 능력 |
| 복잡 (6+ 파일, 아키텍처) | verifier (opus) | 깊은 추론 필요 |
| 보안 관련 | security-reviewer (opus) | 보안 전문성 필요 |

## 금지 사항

- **절대 금지**: 실행 에이전트가 자기 코드를 자기가 최종 검증
- **절대 금지**: "테스트 통과했으니 완료" 선언 (테스트 자체의 품질을 먼저 검증)
- **절대 금지**: 검증 에이전트에게 실행 에이전트의 컨텍스트 공유
- **절대 금지**: FAIL 판정 후 검증 없이 "수정했으니 이번엔 됐을 거야" 선언
